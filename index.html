Introdução

Para que a equipe de qualidade desempenhasse seu trabalho de forma a otimizar tempo e esforços são criados testes, os quais são divididos em níveis e tipos para que cada qual seja utilizado da melhor forma a depender do nível de esforço ou trabalho assegurando o equilíbrio de ênfase e esforço de trabalho em cada nível de teste, sendo assim os níveis de teste são:

Teste Unitário ou de Unidade:

Como o nome já sugere o teste unitário verifica um método, função ou classe de modo isolado, único ou individual. Seu propósito e de evitar erros de homologação na fase de Q.A

Como complemento apresenta-se algumas dicas para realizar bons testes Unitários, sendo estas:

  • Liste os seus testes unitários: Isso ajuda a saber qual teste você está realizando.
  • Testes Unitários precisam ser curtos:O teste unitário precisa ser o mais fechado,unitário e curto possível.
  • Evite testes repetitivos:Cada um dos testes precisam ter um objetivo muito claro e a repetição de testes apenas tomará tempo.
  • Valide seu teste:Faça e refaça seus testes, explore e utilize-os.
  • Não ignore os testes!

Teste de Integração ou componente

(voltar)

Esse tipo de teste é mais comum de ser feito pelo desenvolvedor, mas também pode ser feito pelo Q.A ou Tester.
A diferença entre teste de componente e teste de unidade é que no teste integrado olhamos para duas funções para garantir que não há nenhuma quebra no que foi feito unitariamente e no que está sendo compilado junto.
Testes de componentes são comuns em projetos que envolvem várias pessoas que desenvolvem este software em partes, tem por objetivo encontrar falhas na integração das unidades e é realizada geralmente pelo time de desenvolvedores.
Uma das aplicabilidades pode ser em testes de interface, para garantir e testar as dependências dos componentes.
Mesmo o Q.A não estando envolvido no teste de integração é interessante a sua participação a fim de facilitar a entrega de qualidade para o time.

A fins de economia de tempo, geralmente em algumas empresas o time pode escalar uma pessoa para ficar responsável pelos testes de integração.
Em algumas organizações o time de qualidade pode usar mocks, fakes ou stubs, que são programas emuladores onde simula " Algo fingindo ser outro para que não precise ser este outro", ou seja, um ambiente emulado com as condições necessárias para rodar os componentes a serem testados .

Níveis de teste

Teste de Sistema

Nesta etapa teoricamente o produto já está desenvolvido e pronto para testar as funcionalidades, o objetivo deste nível é analisar que o sistema como um todo funcionará como esperado pelo cliente, encontrando possíveis inconsistências relacionadas aos requisitos, regras de negocio, produto e etc.

Dicas de como realizar bons testes de sistema:
  1. Usar ferramentas eficazes para controle de testes.
  2. Testes de sistema precisam ser feitos em um ambiente especifico para testes, ambiente de homologação.
  3. O QA precisa de um material para consulta.
  4. Tenha uma equipe multidisciplinar e valorize o trabalho em equipe.
  5. Tenha testes constantes no sentido de continuidade
  • Testes funcionais: Tem por objetivo garantir o fluxo do produto dentro do universo perfeito, ou seja, que as funcionalidades que deveriam operar de maneira X realmente operem de maneira X e não Y.
  • Testes não funcionais: De oposto aos testes funcionais, os testes não funcionais tem por objetivo testar o universo imperfeito do produto, testar o que o produto não deve fazer, procurando por bugs, erros de funcionalidade e texto.
  • Teste de Usabilidade: Nesta camada é verificado se o produto é realmente útil para o objetivo ao qual foi projetado, afinal de nada adiante vender um carro sem as rodas, o produto continuaria sendo um carro porém inútil para a função que foi projetado.
  • Teste de Performance: Esta camada verifica o desempenho geral do produto como capacidade , velocidade e afins.
  • Teste de Regreção: É um aglomerado dos testes anteriores, porem com escopo reduzido, e tem por objetivo garantir a qualidade em uma nova versão do produto, sua aplicação é feita sempre que tenha atualização do produto garantindo a funcionalidade dos recursos antigos e também dos novos que entrarão em operação.
  • Teste de Segurança: Garantir a segurança do produto forçando quebra de informações sigilosas e corrigindo falhas de segurança

Processos do teste de sistema

  1. Mapeamento: Levantar casos de testes, o que é preciso ser testado nesse produto e construir cenários com base nos requisitos requisitados.
  2. Modelagem: Seguido do mapeamento já com os casos de testes prontos e filtrados, na modelagem é descrito a forma, o passo a passo e o resultado esperado do teste, a modelagem descreve como o teste deverá ser realizado como uma receita de bolo.
  3. Execução: Se a modelagem é o ato de escrever a receita, a Execução e o ato de fazer o bolo, colocar o programa para funcionar e verificar se funcionará como o planejado.
  4. Teste de aceite: Assim como o teste de Sistema o teste de Aceite é voltado para as funcionalidades do produto porém com um escopo bem menor. Voltado para o usuário final ou PO, este nível visa garantir que o produto é o que se espera que ele seja.

    5. Dicas para teste de aceite:

    • Entenda responsabilidades:
    • O cliente precisa saber que esta fase é para que ele mesmo teste o produto, evitando problemas e não volte de produção com problemas.
    • O cliente pode utilizar os testes do QA.
    • Escreva BDDs com os clientes.
    • Realize testes de aceite quando encontrar bugs, melhorias e atts.

Sêquência e cronometragem dos níveis de teste:

Tradicionalmente pensam que o teste de unidade é implementado logo na interação como primeiro teste, todas as unidades requeridas para serem transmitidas antes dos estágios subsequentes são conduzidas. No entanto, em um processo de desenvolvimento interativo essa abordagem é uma regra geral inapropriada, uma abordagem melhor é identificar os testes de unidade, de integração e do sistema que ofereçam mais potencial para a localização de erros e em seguida implementar, executando com base em uma combinação do maior risco e do ambiente de suporte.

Ferramentas de teste

Primeiramente, ferramentas de teste e testes manuais são objetos distintos, já que podemos optar por testes manualmente ou utilizar de alguma ferramenta para nos auxiliar, sendo que estas ferramentas podem ser existentes no mercado ou criadas pelo próprio Q.A, tendo também a possibilidade de automatizar as mesmas.

Sendo assim, cada ferramenta de teste possui sua própria característica, proposta e padrão o que a torna pouco personalizável e em alguns casos dependendo do seu uso pode ser ineficaz, a principal vantagem no uso dessas ferramentas esta na sua agilidade.

Em contra partida, os testes manuais são altamente personalizáveis, e por isso são seguros já que o utilizador os conhece de ponta a ponta entrada e saída do processo. Porém sua maior desvantagem está no longo tempo gasto, já que este tipo de processo costuma ser muito lento, alem de que o teste pode simplesmente não funcionar dependendo do conhecimento e experiência do operador.

Aprofundando no mundo dos testes de software observa-se que deveríamos imaginar os testes como algo alem de automatizados ou manuais, já que se observarmos ambos individualmente chegaremos a conclusão de que sim, são importantes, mas totalmente substituíveis, entretanto, se observarmos todo o universo de testes de uma forma ampla de conhecimento sendo os testes manuais uma bolha bem pequena dentro do mundo dos testes de software, ampliamos a maneira que enxergamos nosso trabalho, e o que era teste manual agora passa a ser uma verificação manual e podemos facilmente automatizar estes processos pois teoricamente um teste automatizado é capaz de substituir um teste manual.

Conhecendo algumas ferramentas:

As ferramentas de testes de software são inúmeras no mercado, sendo que cada uma realiza testes em um determinado ponto no software, um exemplo de ferramenta é o Cucumber, que realiza execução de cenários e pode utilizar outras ferramentas como o Calabash ou Capybara para automação.

Muitas outras ferramentas existem, algumas delas são:

  • PhantosJS: Ferramenta para automação web porem já descontinuada.
  • SoapUI: Ferramenta de Web service
  • Protractor: Primeira ferramenta para testes com Java Script
  • Appium, MonkeyUI: Automação mobille
  • CasperJS: Automação de Java Script
  • JMeter: Teste de Performance
  • WireMock: Para criação de mocks
  • JUnit, NUnit, Jest, PyTest, C Unit, Kotlin Test: Para teste de Unidade
  • Galling: Para teste de Performance
  • Postman, JetPackt, Karate, RestLet: Para microservices
  • Serenity BDD: Para multiplataformas
  • Apion e Expresso: Para mobile
  • Selenium: Mais popular para testes automatizados
